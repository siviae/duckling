# Duckling v4 — Design Document

> **Duckling** is a data landing pipeline. It consumes Kafka topics produced by information systems (IS), buffers micro-batches as Parquet in DuckLake (MinIO + Postgres catalog), and exports compacted data to Iceberg for analytics via Trino and Spark. It is fully self-hostable and scales horizontally on Kubernetes.

## 1. Context & Goals

### Background

Multiple information systems (IS) produce data to Kafka topics. This data must be landed into the company data platform in a governed, scalable, and self-hostable way.

Key constraints:

- Micro-batch flush interval: **30 seconds minimum, 5 minutes maximum** (configurable per topic)
- Flush also triggered when **> 50% of topic lag is consumed** (see §4)
- Many IS, many topics, one shared infrastructure
- Schema registry: **Apicurio** (BACKWARD compatibility enforced; renames prohibited; NONE compatibility blocked)
- Company stack: **Kotlin**, Spark for heavy lifting
- Must be **fully self-hostable** — no managed cloud services required
- Scales **horizontally on Kubernetes**
- IS enforces its own Kafka topic ownership via ACLs (duckling service account added to topic ACL by the IS)

### Goals

- Land Kafka messages into DuckLake (MinIO + Postgres) with low overhead
- Export compacted Parquet files to Iceberg on a regular cadence
- Enforce schema governance via Apicurio at ingest time
- Retain landed Parquet files until successfully exported to Iceberg, then clean up
- Stop a topic pipeline on schema conflict without affecting other topics
- Be easy to run locally via Docker Compose

### Non-Goals

- Querying landed data before export (no fresh query layer)
- Sub-second streaming
- OLTP-style point updates
- Dead letter queues or message replay
- In-flight SQL transformation of records

## 3. Component Design

### 3.1 Landing Process (Kotlin)

A single process reads `duckling.yaml` and launches one coroutine pipeline per topic under a `SupervisorJob`. A failure in one topic stops only that coroutine; all others continue.

#### Coroutine-Per-Topic

```
CoroutineScope(SupervisorJob + Dispatchers.Default)
  ├── topicPipeline("is_a.orders")
  ├── topicPipeline("is_a.events")
  └── topicPipeline("is_a.metrics")
```

Each topic pipeline:

```
KafkaConsumer.poll()           (IO dispatcher)
    ↓ Channel<ConsumerRecord>  (capacity: 2 × batch_size)
BatchAccumulator               (Default dispatcher)
    — flush trigger: offset progress or time limit —
    ↓ Channel<VectorSchemaRoot> (capacity: 4)
DuckLakeWriter                 (IO dispatcher)
    ↓
Kafka offset commit
```

#### Schema Validation

On startup, the pipeline checks the topic's Apicurio compatibility mode. If it is not BACKWARD, the coroutine stops immediately.

On each batch, before writing, the pipeline validates:
1. All field names are in `[a-zA-Z0-9_]` and not DuckDB reserved words
2. All value types are in the allowed set (see §5)
3. No nested objects or arrays
4. No NaN / Infinity in numeric fields
5. The schema version has not introduced a rename or a non-widening type change

If any check fails, the topic coroutine stops and emits `duckling_topic_stopped_total{topic, reason}`. The operator must fix the schema and restart the pod (or just the affected deployment).

#### Zero-Copy Data Path

```
Kafka Avro bytes
    → deserialize with Apicurio Avro deserializer
    → Arrow VectorSchemaRoot  (off-heap RootAllocator / Netty)
    → DuckLake write via DuckDB JDBC  (Arrow C Data Interface, zero copy)
    → Arrow buffers released via close()
```

No SQL transformation is applied. Records are written as-is, with column types determined by the type mapping in §5.

### 3.2 DuckLake Store

#### Postgres Catalog

Owned by the platform team, operated via the Zalando Postgres Operator (HA, failover, read replicas — out of scope). All DuckLake writers share one Postgres instance. Concurrent writes to the same table serialize through Postgres transactions; writes to different tables are independent.

Custom table for export tracking:

```sql
CREATE TABLE ducklake_export_state (
    table_name              TEXT PRIMARY KEY,
    last_exported_snapshot  BIGINT,
    last_export_time        TIMESTAMPTZ,
    export_enabled          BOOLEAN DEFAULT true,
    export_blocked          BOOLEAN DEFAULT false,
    export_blocked_reason   TEXT
);
```

#### MinIO Layout

One MinIO bucket per IS:

```
s3://{is_name}/
└── ducklake/
    └── {table_name}/
        ├── ducklake-{uuid}.parquet   (one per flush, < 1MB typical)
        └── ...
```

#### File Lifecycle

Files are retained until the Spark exporter confirms a successful Iceberg write. After confirmation:

```sql
-- Record export progress
UPDATE ducklake_export_state
SET last_exported_snapshot = :snapshot_id,
    last_export_time = NOW()
WHERE table_name = :table_name;

-- Expire and clean up Parquet files for exported snapshots only
CALL ducklake_expire_snapshots('ducklake', up_to_snapshot => :snapshot_id);
CALL ducklake_cleanup('ducklake');
```

No data is ever deleted unless it has been confirmed in Iceberg.

### 3.3 Spark Exporter

A PySpark job submitted on a cron schedule (every 5 minutes). Discovers tables ready for export, reads their Parquet files from MinIO, writes compacted data to Iceberg, then triggers DuckLake cleanup.

#### Export Trigger

A table is ready when either:
- Accumulated unexported bytes ≥ 128 MB, OR
- Time since last export ≥ 60 minutes

```sql
SELECT
    f.table_name,
    e.last_exported_snapshot,
    MAX(s.snapshot_id)     AS current_snapshot,
    SUM(f.file_size_bytes) AS accumulated_bytes
FROM ducklake_data_files f
JOIN ducklake_snapshots s USING (snapshot_id)
LEFT JOIN ducklake_export_state e USING (table_name)
WHERE
    (e.last_exported_snapshot IS NULL OR s.snapshot_id > e.last_exported_snapshot)
    AND COALESCE(e.export_enabled, true) = true
    AND COALESCE(e.export_blocked, false) = false
GROUP BY f.table_name, e.last_exported_snapshot
HAVING
    SUM(f.file_size_bytes) >= 128 * 1024 * 1024
    OR EXTRACT(EPOCH FROM NOW() - MIN(s.commit_time)) / 60 >= 60
```

#### Per-Executor Export

```python
def export_table(candidate):
    conn = duckdb.connect()
    conn.execute("ATTACH 'ducklake:postgres:...' AS ducklake (READ_ONLY)")

    files = conn.execute("""
        SELECT file_path FROM ducklake_data_files
        WHERE table_name = ? AND snapshot_id > ?
    """, [candidate.table_name, candidate.last_exported_snapshot or 0]).fetchall()

    df = spark.read \
        .option("mergeSchema", "true") \
        .parquet(*[f[0] for f in files])

    sync_iceberg_schema(candidate.table_name)  # see §6

    df.writeTo(f"iceberg.{candidate.table_name}") \
      .option("write.target-file-size-bytes", str(128 * 1024 * 1024)) \
      .append()

    mark_exported_and_cleanup(candidate)

spark.sparkContext.parallelize(candidates).foreach(export_table)
```

If a table's export fails, it is skipped and retried next run. Other tables are unaffected. Cluster sizing is handled by the platform team and is out of scope.

### 3.4 Iceberg

- Full historical record, 128MB compacted Parquet files, partitioned by time
- **REST catalog: Lakekeeper** — chosen because it is self-hostable, supports both local dev (Docker) and production Kubernetes deployment, and implements the standard Iceberg REST catalog spec. Any Iceberg REST-compatible client (Spark, Trino, DuckDB) connects to it without vendor lock-in.
- Data files stored in the same MinIO bucket as DuckLake, under a separate prefix (`iceberg/`)
- Schema tracked by field ID — renames in Iceberg do not break reads of old files

## 4. Flush Strategy

Each topic coroutine independently decides when to flush its accumulated batch to DuckLake. Two triggers, whichever fires first:

### Trigger 1 — Offset Progress

Every `metadata_poll_interval_seconds` (default: 10 s), the coroutine fetches partition high watermarks from Kafka and computes, per partition:

```
progress = (current_offset − batch_start_offset) / (high_watermark − batch_start_offset)
```

If `progress > offset_progress_flush_threshold` (default: 0.5) for any partition — flush immediately. This keeps file sizes bounded during catch-up replay and prevents unbounded in-memory accumulation.

### Trigger 2 — Time Limit

Flush at most every `min_flush_interval_seconds` (default: 30 s) and at least every `max_flush_interval_seconds` (default: 300 s). A flush triggered by offset progress resets the time counter.

### Summary

| Parameter | Default | Description |
|---|---|---|
| `min_flush_interval_seconds` | 30 | Never flush more often than this |
| `max_flush_interval_seconds` | 300 | Always flush at least this often |
| `metadata_poll_interval_seconds` | 10 | How often to poll partition high watermarks |
| `offset_progress_flush_threshold` | 0.5 | Flush if this fraction of current lag consumed |

## 5. Type System

Duckling defines a strict, flat mapping from JSON value types through the storage and query stack. The mapping uses the lowest common denominator across DuckDB, Parquet, Iceberg, and Trino.

### Allowed Types

| JSON value type | Constraint | DuckDB | Parquet | Iceberg | Trino |
|---|---|---|---|---|---|
| `string` | Any valid UTF-8 | `VARCHAR` | `BYTE_ARRAY` (UTF8) | `string` | `VARCHAR` |
| `number` — integer | Fits in `[-2^63, 2^63−1]` | `BIGINT` | `INT64` | `long` | `BIGINT` |
| `number` — float | IEEE 754 double, finite | `DOUBLE` | `DOUBLE` | `double` | `DOUBLE` |
| `boolean` | `true` / `false` | `BOOLEAN` | `BOOLEAN` | `boolean` | `BOOLEAN` |
| `null` | Value of any nullable field | nullable | optional | optional | nullable |

All columns are nullable. A missing key in a JSON object is treated as `null`.

### Forbidden Constructs

Messages containing any of the following are rejected and cause the topic coroutine to stop:

| Construct | Reason |
|---|---|
| `object` value (nested JSON) | No flat equivalent in Parquet/Iceberg |
| `array` value | Not representable in the type system |
| Integer outside `[-2^63, 2^63−1]` | BIGINT overflow |
| Float `NaN`, `Infinity`, `-Infinity` | Not valid in Parquet DOUBLE |
| Field name with characters outside `[a-zA-Z0-9_]` | Not portable across all systems |
| Field name that is a DuckDB reserved keyword | Breaks generated DDL/DML |

### Permitted Type Widening (Schema Evolution Only)

| From | To | DuckDB | Parquet | Iceberg |
|---|---|---|---|---|
| `BIGINT` | `DOUBLE` | `ALTER COLUMN … TYPE DOUBLE` | New files use `DOUBLE`; old `INT64` files read as double | `update_column` |

All other type changes are prohibited (see §6).

## 6. Schema Evolution

Apicurio is the single source of truth. Duckling does not re-implement compatibility logic — it trusts Apicurio registration and enforces the additional rules below.

### Rules

| Rule | Enforcement |
|---|---|
| Compatibility mode must be BACKWARD | Checked on startup; coroutine stops if not |
| Renames prohibited | Avro `aliases` detected and treated as conflict; coroutine stops |
| Adding a nullable column | Allowed; `ALTER TABLE … ADD COLUMN IF NOT EXISTS … DEFAULT NULL` |
| Removing a column | Allowed; old files retain the column; new files omit it |
| Widening `BIGINT → DOUBLE` | Allowed; `ALTER TABLE … ALTER COLUMN … TYPE DOUBLE` |
| Any other type change | Conflict; coroutine stops |
| NONE compatibility mode | Blocked; coroutine stops on startup |

### Evolution at Each Layer

**DuckLake** — on each batch, the pipeline diffs the Apicurio schema against the current DuckLake table columns and applies only safe DDL:

```kotlin
diff.addedColumns.forEach { col ->
    conn.execute("ALTER TABLE $table ADD COLUMN IF NOT EXISTS ${col.name} ${col.duckType} DEFAULT NULL")
}
diff.widenedColumns.forEach { col ->
    conn.execute("ALTER TABLE $table ALTER COLUMN ${col.name} TYPE ${col.newDuckType}")
}
// Removals: no DDL — new Parquet files simply omit the column
```

**Parquet** — self-describing; DuckDB and Spark handle column projection and null-fill across old and new files transparently via `mergeSchema = true`.

**Iceberg** — Spark exporter syncs before each write:

```python
with iceberg_table.update_schema() as update:
    for col in diff.added_columns:
        update.add_column(col.name, col.iceberg_type)
    for col in diff.widened_columns:
        update.update_column(col.name, col.new_iceberg_type)
    # Removed columns: never dropped from Iceberg schema
```

```yaml
  metrics:

    - name: is_a.orders
      group_id: duckling-is_a-orders
      table: is_a.orders

      max_flush_interval_seconds: 300
      metadata_poll_interval_seconds: 10
      offset_progress_flush_threshold: 0.5

    - name: is_a.events
      group_id: duckling-is_a-events
      table: is_a.events
```

### Field Reference

| Field | Required | Default | Description |
|---|---|---|---|
| `kafka.brokers` | yes | — | Kafka bootstrap servers |
| `kafka.schema_registry` | yes | — | Apicurio base URL |
| `ducklake.catalog_url` | yes | — | Postgres JDBC URL for DuckLake catalog |
| `metrics.prometheus.port` | no | disabled | Prometheus scrape port |
| `topics[].name` | yes | — | Kafka topic name |
| `topics[].group_id` | yes | — | Kafka consumer group ID |
| `topics[].table` | yes | — | Target table (`is_name.table_name`) |
| `topics[].min_flush_interval_seconds` | no | 30 | Minimum seconds between flushes |
| `topics[].max_flush_interval_seconds` | no | 300 | Maximum seconds between flushes |
| `topics[].metadata_poll_interval_seconds` | no | 10 | Partition high-watermark poll frequency |
| `topics[].offset_progress_flush_threshold` | no | 0.5 | Flush when this fraction of lag consumed |

Environment variables are substituted via `${VAR}`. Missing variables cause a startup failure listing all unresolved names.

The process accepts `--config <path>` (default: `duckling.yaml` in the working directory). Configuration is not reloaded at runtime — a restart is required to pick up changes. In Kubernetes, configuration is delivered via an init container writing the file to a shared volume.
