# Duckling v5 — Adaptive Memory Management

> Addendum to v4. All other design decisions remain unchanged.

---

## Problem

In v4, `max_batch_bytes` and `max_flush_interval_seconds` are static per-topic config values. This creates two failure modes:

1. **High throughput** — a topic produces faster than expected. The static interval is too long; the batch grows past the available heap share before a flush fires.
2. **Many topics** — adding more topics to one process shrinks each topic's fair heap share, but the static limits don't know this. Total batch memory = N × static limit, which can exceed `Xmx`.

The fix: derive the effective limits at runtime from `Xmx` and observed per-topic throughput. Configured values become upper bounds; the adaptive layer tightens them as needed.

---

## Design

### Invariant

At any point in time, total in-flight batch memory across all active topics is bounded by:

```
total_batch_memory ≤ Xmx × safety_factor
```

`safety_factor` (default `0.65`) reserves the remaining heap for GC, DuckDB, and stack frames.

### Per-topic budget

```
usable_heap       = Xmx × safety_factor
per_topic_budget  = usable_heap / num_active_topics
effective_max_batch_bytes = min(per_topic_budget, config.max_batch_bytes)
```

`num_active_topics` counts only topics whose coroutines are currently running. When a topic stops (schema conflict), the budget rebalances across survivors automatically.

### Adaptive flush interval

The flush interval is derived from how long it takes to fill the budget at the current throughput:

```
fill_time_seconds        = effective_max_batch_bytes / ema_bytes_per_sec
effective_max_interval   = clamp(fill_time_seconds,
                               config.min_flush_interval_seconds,
                               config.max_flush_interval_seconds)
```

If throughput is unknown (first cycle), `effective_max_interval` falls back to `config.max_flush_interval_seconds`.

### Throughput estimator

Each topic maintains an exponential moving average (EMA) of bytes-per-second, updated after every flush:

```
sample   = bytes_in_flushed_batch / elapsed_seconds_since_last_flush
ema_new  = α × sample + (1 − α) × ema_old      (α = 0.3 default)
```

EMA alpha of 0.3 weights recent observations moderately. A lower alpha reacts more slowly to spikes (safer but slower to adapt); a higher alpha reacts faster (tighter memory but more variable flush cadence).

### Byte-based flush trigger

A third flush trigger is added alongside the existing time and offset-progress triggers:

```
shouldFlush = batch.isNotEmpty() &&
    (
        (pastMin && (pastEffectiveMax || flushByOffset))  -- existing triggers
        || batchBytes >= effective_max_batch_bytes         -- new: byte safety valve
    )
```

The byte trigger **bypasses `pastMin`** because it is a safety valve against unexpected throughput spikes. In steady state it rarely fires (the adaptive interval already accounts for throughput), but when it does, it prevents OOM regardless of the minimum interval.

### Budget recomputation

The budget is recomputed from `AdaptiveController.budgetFor()` **after every flush**, not on a timer. This means the effective limits always reflect the latest throughput estimate and the current number of active topics.

## New configuration

### `AppConfig` — new `adaptive` block

```yaml
duckling:
  adaptive:
    safety_factor: 0.65    # fraction of Xmx available for batch accumulation
    ema_alpha: 0.3         # EMA smoothing factor for throughput estimation
```

Both fields are optional with the defaults shown.

### `TopicConfig` — new `max_batch_bytes` field

```yaml
topics:
  - name: is_a.orders
    max_batch_bytes: 67108864   # 64 MB upper bound (default); adaptive may lower this
    ...
```

This is the **ceiling** the adaptive layer will never exceed for this topic. Operators can lower it for topics where small Parquet files are acceptable, or raise it for low-throughput topics where larger files are preferred.

## Metrics

The following gauges are added so operators can observe the adaptive controller's behaviour:

| Metric | Labels | Description |
|---|---|---|
| `duckling_adaptive_max_batch_bytes` | `topic` | Current effective max batch size in bytes |
| `duckling_adaptive_max_interval_seconds` | `topic` | Current effective max flush interval |
| `duckling_adaptive_throughput_bytes_per_sec` | `topic` | EMA throughput estimate |
| `duckling_batch_flushed_bytes` | `topic` | Actual bytes in each flushed batch (histogram) |

## Heap sizing guide

```
Xmx ≥ (N_topics × max_batch_bytes_per_topic) / safety_factor + 512 MB overhead
```

Example: 10 topics, 64 MB max batch, safety factor 0.65:

```
Xmx ≥ (10 × 64 MB) / 0.65 + 512 MB  ≈  1498 MB  →  set -Xmx1536m
```

The adaptive layer handles any topology within this envelope. If throughput is lower than the worst case, flushes happen more often and batches stay smaller — `Xmx` is never fully utilised. If more topics are added than planned, the per-topic budget shrinks and flush intervals tighten rather than OOMing.
