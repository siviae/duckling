# Duckling v3 — Design Document

> **Duckling** is a data landing pipeline. It consumes Kafka topics produced by information systems (IS), buffers micro-batches as Parquet in DuckLake (S3 + Postgres catalog), and exports compacted data to Iceberg for analytics via Trino and Spark. It is fully self-hostable and scales horizontally on Kubernetes.

## 1. Context & Goals

### Background

Multiple information systems (IS) produce data to Kafka topics. This data must be landed into the company data platform in a governed, scalable, and self-hostable way.

Key constraints:

- Micro-batch flush interval: **30 seconds minimum, 5 minutes maximum** (configurable per topic)
- Flush also triggered when **> 50% of topic lag is consumed** (see §4)
- Many IS, many topics, one shared infrastructure
- Schema registry: **Apicurio** (BACKWARD compatibility enforced; renames prohibited; NONE compatibility blocked)
- Company stack: **Kotlin**, Spark for heavy lifting
- Must be **fully self-hostable** (no managed cloud services required)
- Scales **horizontally on Kubernetes**

### Goals

- Land Kafka messages into DuckLake (S3 + Postgres) with low overhead
- Export compacted Parquet files to Iceberg on a regular cadence
- Enforce schema governance via Apicurio at ingest time
- Keep landed data until successfully exported to Iceberg, then clean up
- Stop a topic pipeline on schema conflict without affecting other topics

### Non-Goals

- Fresh/ad-hoc querying of landed data
- Sub-second streaming
- OLTP-style point updates
- Dead letter queues or message replay

## 3. Component Design

### 3.1 Landing Process (Kotlin)

A single process reads `duckling.yaml` and launches one coroutine pipeline per topic. All pipelines run concurrently in the same JVM under a `SupervisorJob` — a failure in one topic stops only that coroutine.

#### Coroutine-Per-Topic

```
CoroutineScope(SupervisorJob + Dispatchers.Default)
  ├── topicPipeline("is_a.orders")
  ├── topicPipeline("is_a.events")
  └── topicPipeline("is_a.metrics")
```

Each topic pipeline:

```
KafkaConsumer.poll()  (IO dispatcher)
    ↓ Channel<ConsumerRecord>  (capacity: batch_size * 2)
BatchAccumulator  (Default dispatcher)
    ↓ Channel<VectorSchemaRoot>  (capacity: 4)
DuckDBHandler — SQL transform  (Default dispatcher)
    ↓ Channel<VectorSchemaRoot>  (capacity: 4)
DuckLakeWriter  (IO dispatcher)
    ↓
Kafka offset commit
```

#### Schema Validation

On each batch, before SQL transformation, the pipeline checks that the Apicurio schema version embedded in each message's Avro header matches the current registered schema. If a schema conflict is detected (NONE compatibility, rename detected, type narrowing):

1. The topic coroutine logs the error and stops
2. A Prometheus metric `duckling_topic_stopped_total{reason="schema_conflict"}` is emitted
3. Other topic coroutines continue unaffected
4. The process does **not** restart the stopped coroutine — operator intervention is required

#### Zero-Copy Data Path

```
Kafka bytes
    → Arrow VectorSchemaRoot  (off-heap RootAllocator / Netty)
    → DuckDB via Arrow C Data Interface  (zero copy)
    → DuckDB result → Arrow  (zero copy)
    → DuckLake Parquet write
    → Arrow buffers released via close()
```

### 3.2 DuckLake Store

#### Postgres Catalog

Owned and operated by the platform team using the Zalando Postgres Operator (HA, failover, read replicas). Each DuckLake installation uses one shared Postgres instance.

Custom table for export tracking:

```sql
CREATE TABLE ducklake_export_state (
    table_name              TEXT PRIMARY KEY,
    last_exported_snapshot  BIGINT,
    last_export_time        TIMESTAMPTZ,
    export_enabled          BOOLEAN DEFAULT true,
    export_blocked          BOOLEAN DEFAULT false,
    export_blocked_reason   TEXT
);
```

#### S3 Layout

One S3 bucket per information system:

```
s3://{is_name}/
└── ducklake/
    └── {table_name}/
        ├── ducklake-{uuid}.parquet   (one per flush, typically < 1MB)
        └── ...
```

#### File Lifecycle

Parquet files are **not deleted on a time schedule**. They are retained until the Spark exporter marks the corresponding snapshots as exported. After a successful Iceberg export, the exporter calls DuckLake cleanup for the exported snapshots:

```sql
-- Mark snapshot as exported (custom state table)
UPDATE ducklake_export_state
SET last_exported_snapshot = :snapshot_id,
    last_export_time = NOW()
WHERE table_name = :table_name;

-- Expire snapshots up to the exported one
CALL ducklake_expire_snapshots('ducklake', up_to_snapshot => :snapshot_id);
CALL ducklake_cleanup('ducklake');
```

This guarantees: **no data is deleted unless it has been successfully written to Iceberg**.

### 3.3 Spark Exporter

Scheduled every 5 minutes. Discovers tables ready for export, reads Parquet from S3, writes compacted data to Iceberg, then triggers DuckLake cleanup.

#### Export Trigger

A table is ready when either:
- Accumulated unexported bytes ≥ 128 MB, OR
- Time since last export ≥ 60 minutes

```sql
SELECT
    f.table_name,
    e.last_exported_snapshot,
    MAX(s.snapshot_id)     AS current_snapshot,
    SUM(f.file_size_bytes) AS accumulated_bytes
FROM ducklake_data_files f
JOIN ducklake_snapshots s USING (snapshot_id)
LEFT JOIN ducklake_export_state e USING (table_name)
WHERE
    (e.last_exported_snapshot IS NULL OR s.snapshot_id > e.last_exported_snapshot)
    AND COALESCE(e.export_enabled, true) = true
    AND COALESCE(e.export_blocked, false) = false
GROUP BY f.table_name, e.last_exported_snapshot
HAVING
    SUM(f.file_size_bytes) >= 128 * 1024 * 1024
    OR EXTRACT(EPOCH FROM NOW() - MIN(s.commit_time)) / 60 >= 60
```

#### Per-Executor Export

```python
def export_table(candidate):
    conn = duckdb.connect()
    conn.execute("ATTACH 'ducklake:postgres:...' AS ducklake (READ_ONLY)")

    files = conn.execute("""
        SELECT file_path FROM ducklake_data_files
        WHERE table_name = ? AND snapshot_id > ?
    """, [candidate.table_name, candidate.last_exported_snapshot or 0]).fetchall()

    df = spark.read \
        .option("mergeSchema", "true") \
        .parquet(*[f[0] for f in files])

    # Sync Iceberg schema to latest Apicurio schema before writing
    sync_iceberg_schema(candidate.table_name)

    df.writeTo(f"iceberg.{candidate.table_name}") \
      .option("write.target-file-size-bytes", str(128 * 1024 * 1024)) \
      .append()

    # Cleanup DuckLake only after confirmed Iceberg write
    mark_exported_and_cleanup(candidate)

spark.sparkContext.parallelize(candidates).foreach(export_table)
```

If export of a table fails, that table is skipped and retried in the next run. Other tables are unaffected. Export cluster sizing is handled by the platform team and is out of scope.

### 3.4 Iceberg

- Full historical record, compacted 128MB Parquet files
- Partitioned by time
- Schema tracked by field ID (not name) — field IDs assigned by the Spark exporter on first write
- REST catalog (self-hostable)

**Why not write directly to Iceberg:** at 30-second flushes, direct Iceberg writes produce thousands of tiny files and commits per day, degrading query performance and requiring constant compaction. DuckLake absorbs this; Iceberg receives only properly-sized files.

## 4. Flush Strategy

Each topic coroutine independently decides when to flush its accumulated batch to DuckLake (S3 + Postgres). Two triggers, whichever fires first:

### Trigger 1 — Offset Progress

Every `metadata_poll_interval_seconds` (default: 10s), the coroutine fetches partition metadata from Kafka and computes:

```
progress = (current_offset - batch_start_offset) / (high_watermark - batch_start_offset)
```

If `progress > 0.5` — more than half of the available lag has been consumed since the last flush — flush immediately, regardless of elapsed time.

This prevents the pipeline from buffering indefinitely when there is a large backlog to catch up on. It keeps individual Parquet files small (< 1MB target) even during replay.

### Trigger 2 — Time Limit

Flush at least every `max_flush_interval_seconds` (default: 300s / 5 min), and at most every `min_flush_interval_seconds` (default: 30s). After a flush triggered by offset progress, the time counter resets.

### Flush Interval Constraints

```
min_flush_interval_seconds ≤ actual_flush_interval ≤ max_flush_interval_seconds
```

Defaults:

| Parameter | Default | Notes |
|---|---|---|
| `min_flush_interval_seconds` | 30 | Never flush more often than this |
| `max_flush_interval_seconds` | 300 | Always flush at least this often |
| `metadata_poll_interval_seconds` | 10 | How often to check partition high watermarks |
| `offset_progress_flush_threshold` | 0.5 | Flush if this fraction of lag consumed |

All are configurable per topic in `duckling.yaml`.

## 5. Type System

Duckling defines a strict mapping from JSON value types to DuckDB (in-process), Parquet (storage), Iceberg (catalog), and Trino (query). The lowest common denominator across all four systems governs what is allowed.

### Allowed JSON Value Types

| JSON type | Constraint | DuckDB type | Parquet type | Iceberg type | Trino type |
|---|---|---|---|---|---|
| `string` | Any UTF-8 | `VARCHAR` | `BYTE_ARRAY` (UTF8) | `string` | `VARCHAR` |
| `number` (integer) | Fits in signed 64-bit integer | `BIGINT` | `INT64` | `long` | `BIGINT` |
| `number` (float) | IEEE 754 double, finite | `DOUBLE` | `DOUBLE` | `double` | `DOUBLE` |
| `boolean` | `true` / `false` | `BOOLEAN` | `BOOLEAN` | `boolean` | `BOOLEAN` |
| `null` | Only as value of a nullable field | nullable modifier on any above | optional modifier | optional modifier | nullable |

All top-level fields in the JSON message map to columns in the target table. All columns are **nullable** — a missing key in a JSON object is treated as `null`.

### Forbidden JSON Constructs

The following cause the message to be rejected and the topic pipeline to stop (same behaviour as a schema conflict — operator intervention required):

| Construct | Reason |
|---|---|
| `object` value (nested) | No equivalent in flat Parquet/Iceberg schema |
| `array` value | Not representable in the type system |
| Integer outside `[-2^63, 2^63 - 1]` | Overflow in BIGINT |
| Float `NaN`, `Infinity`, `-Infinity` | Not supported by Parquet DOUBLE |
| Field name that is a DuckDB reserved keyword | Would break generated SQL |
| Field name containing characters outside `[a-zA-Z0-9_]` | Not portable across all systems |

### Type Promotion (Schema Evolution)

The only permitted type change across schema versions is widening within the same column family:

| Allowed widening | DuckDB | Parquet | Iceberg |
|---|---|---|---|
| integer → double | `BIGINT` → `DOUBLE` | `INT64` → `DOUBLE` | `long` → `double` |

All other changes (narrowing, type change across families, rename) are **prohibited** and cause the topic pipeline to stop.

## 6. Schema Evolution

Apicurio is the single source of truth. Duckling trusts Apicurio registration — it does not re-implement compatibility logic.

### Rules Enforced

| Rule | Enforcement |
|---|---|
| BACKWARD compatibility required | Pipeline checks topic compatibility mode on startup; stops if NONE or FORWARD |
| Renames prohibited | Pipeline detects Avro `aliases`; treats any rename as a conflict and stops |
| Adding a nullable column | Allowed; DuckLake `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ... DEFAULT NULL` |
| Removing a column | Allowed; old Parquet files still contain it, new files omit it; both readable |
| Widening integer → double | Allowed; DuckLake `ALTER COLUMN ... TYPE DOUBLE` |
| Any other type change | Conflict; pipeline stops |

### Evolution at Each Layer

**DuckLake** — on each batch, the pipeline fetches the latest Apicurio schema and diffs it against the current DuckLake table schema. If drift is detected:

```kotlin
diff.addedColumns.forEach { col ->
    conn.execute("ALTER TABLE $table ADD COLUMN IF NOT EXISTS ${col.name} ${col.duckType} DEFAULT NULL")
}
diff.widenedColumns.forEach { col ->
    conn.execute("ALTER TABLE $table ALTER COLUMN ${col.name} TYPE ${col.newDuckType}")
}
// Removals: no ALTER needed — new Parquet files simply omit the column
```

**Parquet** — self-describing; old and new files coexist in S3 with differing column sets. DuckDB and Spark handle column projection and null-fill transparently.

**Iceberg** — Spark exporter syncs schema before each write:

```python
with iceberg_table.update_schema() as update:
    for col in diff.added_columns:
        update.add_column(col.name, col.iceberg_type)
    for col in diff.widened_columns:
        update.update_column(col.name, col.new_iceberg_type)
    # Removals: never dropped from Iceberg (old files still reference them)
```

## 7. YAML Configuration

### Full Example

```yaml
duckling:
  kafka:
    brokers: "${KAFKA_BROKERS}"
    schema_registry: "${APICURIO_URL}"

  ducklake:
    catalog_url: "${DUCKLAKE_CATALOG_POSTGRES_URL}"

  metrics:
    prometheus:
      port: 9090

  topics:
    - name: is_a.orders
      group_id: duckling-is_a-orders
      table: is_a.orders
      sql: |
        SELECT order_id, customer_id, region, amount, event_time
        FROM input
        WHERE amount > 0
      min_flush_interval_seconds: 30
      max_flush_interval_seconds: 300
      metadata_poll_interval_seconds: 10
      offset_progress_flush_threshold: 0.5

    - name: is_a.events
      group_id: duckling-is_a-events
      table: is_a.events
      sql: "SELECT * FROM input"
```

### Field Reference

| Field | Required | Default | Description |
|---|---|---|---|
| `kafka.brokers` | yes | — | Kafka bootstrap servers |
| `kafka.schema_registry` | yes | — | Apicurio base URL |
| `ducklake.catalog_url` | yes | — | Postgres JDBC URL for DuckLake catalog |
| `metrics.prometheus.port` | no | disabled | Prometheus scrape port |
| `topics[].name` | yes | — | Kafka topic name |
| `topics[].group_id` | yes | — | Kafka consumer group ID |
| `topics[].table` | yes | — | Target table (`is.table`) |
| `topics[].sql` | yes | — | SQL transform; source table is always `input` |
| `topics[].min_flush_interval_seconds` | no | 30 | Minimum seconds between flushes |
| `topics[].max_flush_interval_seconds` | no | 300 | Maximum seconds between flushes |
| `topics[].metadata_poll_interval_seconds` | no | 10 | How often to poll partition high watermarks |
| `topics[].offset_progress_flush_threshold` | no | 0.5 | Flush if this fraction of current lag consumed |

Environment variables are substituted using `${VAR}` syntax. Missing variables cause a startup failure listing all unresolved names.

The process accepts `--config <path>` (default: `duckling.yaml` in the working directory).
